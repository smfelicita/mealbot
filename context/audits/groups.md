# Groups — аудит

## 1. Текущее состояние

Два типа групп: `FAMILY` и `REGULAR`.

**FAMILY:**
- Одна на пользователя, максимум 10 участников
- Общий холодильник: при вступлении личные FridgeItems мигрируют (groupId проставляется)
- При выходе/исключении — FridgeItems возвращаются (groupId=null)
- При удалении FAMILY — все участники получают продукты обратно

**REGULAR:**
- До 2 на пользователя, максимум 1000 участников
- Только общие рецепты, холодильник отдельный
- **Временно отключены в UI** (backend отдаёт 503/400 на создание)

**Join-коды:** 8 символов, TTL 7 дней, только для REGULAR (и только owner видит).
**Инвайты по email:** one-time token, TTL 7 дней, rate limit 3/день/email, 10/день/sender.

## 2. Найденные проблемы

| Проблема | Приоритет |
|---|---|
| Миграция FridgeItems не была транзакционной | critical |
| Kick участника не возвращал его FridgeItems | high |
| Инвайт: не проверялся email получателя при accept | high |
| REGULAR группы открыты всем — нет ограничения | medium |

## 3. Решения

- Вся миграция холодильника — в `prisma.$transaction`
- Kick и leave — возвращают FridgeItems транзакционно
- Инвайт accept — проверка `invite.email === user.email`
- REGULAR группы — скрыты в UI, backend блокирует создание (до решения о монетизации)

## 4. Реализация

**Файлы:**
- `backend/src/routes/groups.js` — CRUD, join, leave, kick, join-code
- `backend/src/routes/invites.js` — invite flow

Хелпер `getFamilyGroupId(userId)` — во всех роутах, где нужен общий холодильник.

## 5. Открытые вопросы

- Бизнес-решение: когда и как открывать REGULAR группы (платно? бесплатно?)
- Уведомления при вступлении нового участника в FAMILY

## 6. Договорённости

- Миграция холодильника — только через транзакцию, не по отдельности
- `getFamilyGroupId()` — единственный способ получить ID семейной группы
- REGULAR группы не включать без явного решения
- Лимиты (FAMILY: 1 группа/10 участников, REGULAR: 2/1000) не менять без обсуждения
